Feature 03
Stage A / B Split + Parallel Deploy Architecture
Deployment Pipeline — Why Split: Parallelization
Stage A — L1-Independent
Infrastructure Provisioning
  • EKS cluster creation
  • EFS, NLB, ALB
  • Helm chart initial install
  • DNS / TLS configuration
L1 Contracts — No L2 RPC Needed
Contract Deployment
  • Deploy.s.sol execution
  • DGF / OptimismPortal
  • DelayedWETH etc. initialization
  • Address collection
→
errgroup
Stage B — Requires L1 Addresses
Chain Node Startup
  • Contract address injection
  • Genesis generation + sync
  • op-geth / op-node startup
  • Anchor State submission
Stage A + L1 contracts run in parallel · Stage B runs sequentially after both complete ~6 min saved
Parallelization Effect
L1 init functions have no L2 RPC dependency — verified at code level. No nonce conflicts. ~6 min saved via errgroup concurrent execution.
Additional Benefit of Splitting
On failure, only Stage B needs restarting. EKS provisioning (Stage A) does not need to rerun. Debugging cost drastically reduced.
